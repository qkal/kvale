# Kvale v0.1.2 — Design Spec

**Date:** 2026-03-30
**Authors:** Kal (Qkal) + Claude (Anthropic)
**Status:** Approved — ready for implementation

---

## Overview

Kvale v0.1.2 is the production-hardening release. v0.1.1 shipped a complete data layer — mutations, reactive keys, deduplication, invalidation, gcTime, select, prefetch. v0.1.2 adds the three things that make Kvale safe to recommend for real production SvelteKit apps: SSR support via `dehydrate`/`rehydrate`, robust error handling (global hook, conditional retry, timeouts), and an observability event bus. No new data-fetching primitives — only hardening.

---

## Scope

### In v0.1.2

**SSR:**
1. `cache.dehydrate()` — serialize cache state to a JSON-safe snapshot
2. `cache.rehydrate(state)` — seed client cache from snapshot before first render

**Error handling:**
3. Global `onError` hook on `CacheConfig` — fires once per final failure (query + mutation)
4. `retry` accepts `(failureCount: number, error: Error) => boolean` in addition to `number`
5. `timeout` on `CacheConfig` and `QueryConfig` — aborts fetch after N ms

**Observability:**
6. `onEvent` hook on `CacheConfig` — typed event bus for all cache lifecycle events
7. `CacheEvent` — exported discriminated union type

### Deferred to v0.1.3

- **Infinite / paginated queries** — `cache.infiniteQuery()` with `fetchNextPage`, `hasNextPage`
- **Structural sharing** — deep equality to prevent re-renders when data is unchanged
- **Query observers** — `cache.subscribe(key, callback)` public API
- **Devtools package** — `kvale/devtools` cache inspector UI (prep done via `entries()`)

### Key Architectural Decisions

**`src/core/ssr.ts` is pure TypeScript.**
It reads `CacheStore` entries via the existing `entries()` iterator. No Svelte import. Importable in `+page.server.ts` without the Svelte compiler.

**`timeout` is a `Promise.race` inside `QueryRunner.fetchWithRetry`.**
Each individual fetch attempt (including retries) gets its own fresh timeout — not a shared total. When the timeout wins the race, it calls `controller.abort()` and throws `new Error('Request timed out')`. This re-uses the existing abort path — no special handling needed downstream.

**`retry` number is normalized to a function at `QueryRunner` construction.**
`retry: 3` becomes `(count) => count < 3`. All internal retry logic works with the function form only. Backwards-compatible — existing `number` configs are unchanged in behavior.

**`onError` fires after `onSettled`, only on final failure.**
For queries: fires after all retries are exhausted, after the runner sets `status: 'error'`. For mutations: fires via the `MutationRunner`, which receives `onError`/`onEvent` from `createCache`'s resolved config, forwarded through `cache.mutate()`. Mutation fires with `key: []`. If `onError` itself throws, the error is silently swallowed.

**`onEvent` fires on every cache lifecycle event, including intermediate retry failures.**
`fetch:error` fires on every failed attempt with the current `failureCount`. `onError` fires only once at the end. This lets `onEvent` track retry telemetry while `onError` handles user-facing responses.

**`rehydrate` is additive.**
It only writes entries not already present in the client cache. Calling it twice is safe. It preserves the original `timestamp` from the server so stale-while-revalidate works correctly — a 25-second-old entry rehydrated with `staleTime: 30_000` will revalidate in 5 seconds, not 30.

## API

### `CacheConfig` — 3 new options

```ts
interface CacheConfig {
  staleTime: number;              // existing — default: 30_000
  retry: number | ((failureCount: number, error: Error) => boolean); // CHANGED — accepts function
  refetchOnWindowFocus: boolean;  // existing — default: true
  refetchOnReconnect: boolean;    // existing — default: true
  persist?: Storage;              // existing — default: undefined
  gcTime: number;                 // existing — default: 300_000
  timeout?: number;               // NEW — default: undefined (no timeout)
  onError?: (error: Error, key: string[]) => void; // NEW — fires on final failure
  onEvent?: (event: CacheEvent) => void;           // NEW — fires on all lifecycle events
}
```

### `QueryConfig` — 1 new option

```ts
interface QueryConfig<T, U = T> {
  key: string | unknown[] | (() => string | unknown[]); // existing
  fn: (signal: AbortSignal) => Promise<T>;              // existing
  staleTime?: number;            // existing
  refetchInterval?: number;      // existing
  enabled?: boolean | (() => boolean); // existing
  keepPreviousData?: boolean;    // existing
  select?: (data: T) => U;       // existing
  retry?: number | ((failureCount: number, error: Error) => boolean); // CHANGED — accepts function
  timeout?: number;              // NEW — overrides CacheConfig.timeout for this query
}
```

### `CacheEvent` — new exported type

```ts
export type CacheEvent =
  | { type: 'fetch:start';   key: string[] }
  | { type: 'fetch:success'; key: string[]; duration: number }
  | { type: 'fetch:error';   key: string[]; error: Error; failureCount: number }
  | { type: 'invalidate';    key: string[]; matchedKeys: string[][] }
  | { type: 'set';           key: string[] }
  | { type: 'gc';            key: string[] }
  | { type: 'rehydrate';     keys: string[][] }
```

> Mutation events (`mutation:start`, `mutation:error`, etc.) are deferred to v0.1.3 — only `onError` reaches mutations in this release.

### `DehydratedState` — new exported interface

```ts
export interface DehydratedState {
  entries: Array<{
    key: string;       // serialized cache key
    data: unknown;     // raw T — never transformed by select
    timestamp: number; // original server-side timestamp
  }>;
}
```

### New cache methods

```ts
// Methods on the object returned by createCache() — access store via closure
cache.dehydrate(): DehydratedState
cache.rehydrate(state: DehydratedState): void
```

`src/core/ssr.ts` contains the pure logic and exports functions that take `CacheStore` directly — `createCache()` wraps them. This avoids circular imports and keeps `ssr.ts` server-safe with no dependency on the cache object shape.

## Data Flows

### SSR with `+page.server.ts`

```
+page.server.ts load()
  → createCache() — fresh per-request instance, no shared state
  → serverCache.prefetch({ key, fn })
  → serverCache.dehydrate() → DehydratedState (JSON-safe)
  → return { dehydrated: state }

+page.svelte (client)
  → cache.rehydrate(data.dehydrated) — called before cache.query()
      → for each entry: if client cache has no entry for key → write entry with original timestamp
      → fires onEvent({ type: 'rehydrate', keys })
  → cache.query({ key, fn }) — cache hit, no loading flash
```

### `onError` + `onEvent` on query failure

```
QueryRunner.fetchWithRetry()
  → fn(signal) rejects
  → onEvent({ type: 'fetch:error', key, error, failureCount }) — fires on EVERY failure
  ├─ failureCount < retry limit → delay → retry
  └─ failureCount >= retry limit
       → runner sets status: 'error'
       → onError(error, key[]) — fires ONCE, after all retries
       → if onError throws → silently swallowed
```

### `timeout` race

```
QueryRunner.fetchWithRetry()
  → controller = new AbortController()
  → timeoutId = setTimeout(() => controller.abort(), timeout)
  → Promise.race([fn(signal), timeoutPromise])
  ├─ fn resolves first → clearTimeout(timeoutId) → success path
  └─ timeout fires first → controller.abort() → fn receives aborted signal
       → throws new Error('Request timed out')
       → normal retry / error path applies
```

### `retry` as function

```
fetchWithRetry catches error at attempt N
  → typeof retry === 'number' → normalized to (count) => count < retry
  → retry(failureCount, error)
  ├─ returns false → no retry, status: 'error', onError fires
  └─ returns true  → delay → next attempt
```

### `onError` for mutations

```
cache.mutate({ fn, onError: mutationOnError, ... })
  → createReactiveMutation(mutationConfig, { onError: cacheOnError, onEvent })
  → MutationRunner receives both mutationConfig.onError and cacheOnError
  → mutation fn rejects, all mutation-level callbacks fire
  → cacheOnError(error, []) — fires after onSettled, key is []
  → if cacheOnError throws → silently swallowed
```

## Canonical SvelteKit SSR Pattern

```ts
// +page.server.ts
import { createCache } from 'kvale';

export async function load() {
  const serverCache = createCache();
  await serverCache.prefetch({ key: 'todos', fn: (signal) => fetchTodos(signal) });
  return { dehydrated: serverCache.dehydrate() };
}
```

```svelte
<!-- +page.svelte -->
<script lang="ts">
  import { cache } from '$lib/cache';

  let { data } = $props();
  cache.rehydrate(data.dehydrated); // must be called before cache.query()

  const todos = cache.query({
    key: 'todos',
    fn: (signal) => fetchTodos(signal),
  });
  // status is immediately 'success' — no loading flash
</script>
```

## Error Handling

| Scenario | Behaviour |
|---|---|
| `fn()` rejects, `retry` is number | Retries N times with exponential backoff. `onError` fires once at the end. |
| `fn()` rejects, `retry` is function returning false | No retry. `onError` fires immediately. |
| `fn()` rejects, `retry` is function returning true | Retries while function returns true. |
| Timeout fires | `controller.abort()` called. Fetch throws `'Request timed out'`. Normal retry path applies. |
| `onError` hook throws | Error silently swallowed — never propagates to the component. |
| `onEvent` hook throws | Error silently swallowed. |
| `rehydrate` called twice | Second call is a no-op per key — additive, never overwrites. |
| `dehydrate` on error entry | Entry excluded from snapshot. |
| `dehydrate` on entry with `data: undefined` | Entry excluded from snapshot. |
| Mutation failure + `cacheOnError` | `cacheOnError(error, [])` fires after `onSettled`. |

### `tests/core/ssr.test.ts` — 8 tests

- `dehydrate` with empty cache → `{ entries: [] }`
- `dehydrate` includes fresh entries with correct shape
- `dehydrate` includes stale entries (staleness not filtered — re-evaluated client-side)
- `dehydrate` excludes entries with `error !== null`
- `dehydrate` excludes entries with `data === undefined`
- `rehydrate` populates cache entries
- `rehydrate` preserves original timestamps
- `rehydrate` is additive — existing entries not overwritten
- `rehydrate` calling twice is safe (idempotent)
- `rehydrate` fires `onEvent` with `type: 'rehydrate'` and correct `keys`

- `timeout` per-query overrides global `CacheConfig.timeout`
- `onError` fires once after all retries exhausted, not on intermediate failures
- `onError` receives correct `error` and `key[]`

## Constraints (Non-Negotiable, Carried Forward)

- Zero `dependencies` in `package.json` — only `peerDependencies: { svelte: "^5.25.0" }`
- `src/core/` never imports from `'svelte'`
- `src/core/ssr.ts` never imports from `'svelte'` — must be server-safe
- No `QueryClientProvider` wrapper component
- No Svelte stores (`writable`, `readable`) — runes only
- No `$:` reactive declarations — Svelte 5 runes only
- Bun for all package management and scripts
- TypeScript strict mode, no `any`
